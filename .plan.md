# mindX Multi-Chain Agent Verification & Multi-Stream Inference Plan

## Context

mindX has 171 Solidity contracts in `daio/` including governance (BoardroomExtension, DojoManager, DAIO_Constitution), reputation (AgentReputationRegistry with 0-10000 basis points), identity (IDNFT), and an xmind/ bridge. There is NO Algorand infrastructure yet. The CEO + Seven Soldiers executive board exists in config but is not wired to actual inference consensus. The MultiModelAgent exists but is disconnected from the orchestration hierarchy.

## Phase 1: BONA FIDE Algorand ASA + Agent Multi-Key Identity

**Goal**: Agents earn verification status via BONA FIDE token (Algorand ASA with clawback). Agents hold multiple keys across chains.

### 1a. BONA FIDE ASA Contract (`daio/contracts/algorand/`)
- Create `daio/contracts/algorand/bona_fide.algo.ts` — Algorand TypeScript contract
- ASA with clawback: mindX can revoke BONA FIDE status for misbehaving agents
- Reputation threshold: agent must reach 5000/10000 basis points to earn BONA FIDE
- ASA metadata carries: agent_id, reputation_score, verification_timestamp, issuing_authority
- Clawback conditions: reputation drops below 2500, security violation, consensus revocation

### 1b. Multi-Key Agent Identity (`agents/core/id_manager_agent.py`)
- Extend IDManagerAgent to manage per-agent key bundles:
  - `eth_key`: Ethereum wallet (existing, BANKON vault)
  - `algo_key`: Algorand account (new)
  - `api_credentials`: per-provider API tokens (new)
  - `social_credentials`: external platform tokens (new)
- Store all keys encrypted in BANKON vault under `agent_pk_{agent_id}_{chain}`
- Add `get_agent_key_bundle(agent_id)` returning all chains/credentials
- Create `data/identity/agent_capabilities.json` — maps agent → capabilities, credentials, verification status

### 1c. Agent Verification Scoring (`daio/contracts/algorand/reputation_oracle.algo.ts`)
- Bridge between on-chain BONA FIDE status and off-chain mindX reputation
- Score inputs: task completion rate, peer reviews, improvement campaign success, Gödel decision quality
- Scores stored: on-chain (Algorand box storage) + off-chain (beliefs.json)
- Verification tiers: Unverified → Provisional → Verified → BONA FIDE → Sovereign

## Phase 2: Agent Map & Groupings (`daio/agents/`)

### 2a. Canonical Agent Registry (`daio/agents/agent_map.json`)
- Single source of truth for all agents, their roles, group membership, verification status
- Groups: `core_command` (PYTHAI, SAVANTE, SUNTSU, RAGE), `executive_board` (CEO + Seven Soldiers), `operational` (coordinator, memory, guardian, inference), `learning` (SEA, self_improve, prediction, socratic), `identity` (id_manager, validator)
- Each entry: agent_id, eth_address, algo_address, group, role, bona_fide_status, reputation_score

### 2b. Group Consensus Logic (`daio/agents/group_consensus.py`)
- Implements weighted voting per group
- Uses executive_board.yaml weights (CISO/CRO = 1.2x veto)
- Dissent handling: dissenting opinion recorded, opens exploration branch rather than blocking
- Quorum: 2/3 supermajority per group, 2/3 of groups for cross-group decisions

## Phase 3: Boardroom & Dojo AI Integration (`daio/governance/`)

### 3a. Boardroom Controller (`daio/governance/boardroom.py`)
- Python interface to BoardroomExtension.sol concepts (off-chain first, on-chain later)
- CEO presents directive → Seven Soldiers evaluate in parallel → weighted vote → execute or explore dissent
- Each Soldier queries a different inference provider for its evaluation
- Boardroom session log: who said what, vote weights, final decision, dissent record

### 3b. Dojo Training Rounds (`daio/governance/dojo.py`)
- Maps to DojoManager.sol reputation ranks (Novice → Sage)
- Agents earn reputation by: completing tasks, winning consensus votes, producing quality improvements
- Dojo types: Governance, Treasury, Technical, Compliance, Strategic
- Rank thresholds determine which decisions agent can participate in
- Training mechanism: agent submits improvement proposal → peers score → reputation updates

### 3c. CEO + Seven Soldiers Wiring
- Wire `agents/orchestration/ceo_agent.py` to call boardroom for major decisions
- Each Soldier mapped to an inference provider for diversity:
  - COO → Gemini (operations focus)
  - CFO → Groq (fast financial analysis)
  - CTO → Ollama/qwen3 (local technical eval)
  - CISO → Anthropic (safety analysis) when available
  - CLO → Mistral (compliance)
  - CPO → Together (product reasoning)
  - CRO → vLLM (risk modeling) when available
- Fallback: any unavailable provider → next available in preference order

## Phase 4: Multi-Stream Inference Hierarchy (`llm/multi_stream.py`)

### 4a. Parallel Utterance Streams
- New `MultiStreamInference` class that fires the same prompt to N providers simultaneously
- Configurable strategies: `fastest_wins`, `consensus`, `weighted_vote`, `hierarchical`
- `fastest_wins`: return first response (latency-optimized)
- `consensus`: wait for majority agreement (quality-optimized)
- `weighted_vote`: each response scored by reliability × confidence
- `hierarchical`: CEO decides which strategy based on decision importance

### 4b. Inference Hierarchy (prevents duplicate work)
- Level 1 (instant): Ollama local — fast, cheap, handles routine decisions
- Level 2 (standard): Cloud providers — Gemini, Groq, Mistral for normal operations
- Level 3 (critical): Multi-stream consensus — parallel query for high-stakes decisions
- Level 4 (constitutional): Full boardroom — all Soldiers evaluate, 2/3 supermajority required
- Routing rule: task importance (from BDI agent) determines which level
- Dedup: hash-based request dedup prevents same prompt hitting same provider twice

### 4c. Dissent as Innovation
- When Soldiers disagree, don't discard minority opinion
- Create exploration branch: minority view becomes new BDI goal to investigate
- Track exploration outcomes in improvement journal
- If minority view proves correct, boost that Soldier's reputation
