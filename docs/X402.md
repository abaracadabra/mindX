## What x402 is

[x402.org](https://x402.org) reactivates the long-reserved HTTP 402 status code. A server returns `402 Payment Required` plus an envelope listing one or more `accepts[]` schemes (chain, asset, recipient, amount, resource, nonce). The client picks a scheme, signs a payment authorization with its wallet, and resubmits with `X-PAYMENT: base64(JSON({scheme, network, payload}))`. A facilitator verifies and settles the signed payload on-chain. The server confirms via `X-PAYMENT-RESPONSE`.

The standard is **chain-agnostic by design**. EVM uses ERC-20 + EIP-3009 typed-data signatures. AVM uses an ASA transfer + Ed25519 signatures. mindX's contracts give both chains the **same on-chain receipt shape** so unified indexers can read across.

References:
- Spec: [x402.org](https://x402.org), [x402-whitepaper.pdf](https://x402.org/x402-whitepaper.pdf)
- mindX EVM receipt: [`BankonPaymentRouter.sol`](../daio/contracts/ens/v1/BankonPaymentRouter.sol)
- mindX AVM settler: [`interchain_settler.algo.ts`](../daio/contracts/algorand/interchain_settler.algo.ts)

## Cross-chain receipt — `X402Receipt`

The shipped contracts are intentionally minimal but the production move is a **canonical pair** with one ABI:

```solidity
// EVM (Solidity / Foundry)
function recordX402Receipt(
    bytes32 receiptHash,         // sha256(canonical receipt struct)
    address payer,
    address payee,
    address asset,
    uint256 amount,              // base units (USDC = 6 dp)
    bytes32 resourceHash,        // hash of the paid resource URL
    bytes calldata signature     // EIP-3009 signature of the receipt struct
) external;

event X402ReceiptRecorded(
    bytes32 indexed receiptHash,
    bytes32 indexed resourceHash,
    address payer,
    address payee,
    address asset,
    uint256 amount,
    uint64  chainId,
    uint64  blockOrRound
);
```

```typescript
// Algorand (PuyaTs / AlgoKit)
@arc4.abimethod()
recordX402Receipt(
    receiptHash:  Bytes,         // sha256 of canonical receipt struct (bytes32)
    payer:        Account,       // 58-char base32 (or msig descriptor)
    payee:        Account,
    assetId:      Uint64,
    amount:       Uint64,
    resourceHash: Bytes,         // bytes32
    signature:    Bytes,         // Ed25519 over canonical struct (single or msig-merged)
): void
```

**Compatibility invariants** (the part that matters):
1. **Same `receiptHash` formula** — sha256 over the canonical struct `(payer, payee, asset, amount, resourceHash, nonce)`. A receipt computed off-chain hashes to the same value on either chain.
2. **Same event/log shape** — `X402ReceiptRecorded` on EVM and an arc4-encoded log on Algorand carry the same field set. Indexers join by `receiptHash`.
3. **Idempotent on `receiptHash`** — both chains reject duplicate hashes (`seenReceipt[h] = true` on EVM; `BoxMap` on Algorand).
4. **Multisig is first-class** — see [§ Multisig](#multisig).

The contract is the **attestation layer**; actual value movement is the existing USDC `transferWithAuthorization` (EVM) or `AssetTransferTxn` (AVM). `BankonPaymentRouter` keeps its split/sweep role and is called *upstream* by `X402Receipt` on the EVM side.

Status: design lives in this doc; Foundry + AlgoKit drafts are the next deliverable. Mainnet deploy gated on per-TX human confirmation.

## Multisig

Algorand multisig is native — addresses are derived from `(version, threshold, pkSet)` and a signed transaction merges per-signer signatures. EVM multisig is contract-based (Safe most commonly).

For x402 receipts:

| Feature | EVM | AVM |
|---|---|---|
| Single-sig payer | EIP-3009 sig recover → check `from == payer` | Ed25519 verify → derive address → check matches `payer` |
| **Multisig payer** | Safe-style: payer is a Safe; receipt verified via [EIP-1271](https://eips.ethereum.org/EIPS/eip-1271) `isValidSignature` callback | Native: `payer` is a multisig descriptor; `signature` is the [merged multisig sig](https://developer.algorand.org/docs/clis/goal/clerk/multisig/merge/); contract verifies enough subsigs cleared the threshold |
| **Multisig payee** | `payee` is any address — value lands at the Safe; no special contract logic | `payee` is opted into the ASA — stock multisig account |
| Coordination | OpenZeppelin Governor / Safe Tx Service | The active draft [ARC for multisig signing coordination](https://forum.algorand.co/t/arc-allowing-coordination-of-multisig-signing/7840) — mindX should track and cite once finalised |

`X402Receipt.algo.ts` should accept both the canonical 64-byte single Ed25519 sig and the variable-length merged-multisig sig form, branching on the leading subsig descriptor.

## Wallets and signing paths

mindX is wallet-agnostic. The Algorand-side options today:

### Pera Wallet

[Pera Wallet](https://perawallet.app/) is the most-used Algorand wallet — open-source, self-custodial. The org [refactored into a monorepo](https://github.com/perawallet/pera-react-native) on 2026-05-01 (`pera-react-native`); the legacy `pera-wallet` repo was archived October 2025. Pera's [Connect SDK](https://github.com/perawallet/connect) is the standard browser wallet path.

For x402, Pera surfaces signing via WalletConnect-style request; mindX's `AgenticPlace/api/x402_avm.ts` uses the published [`@x402-avm/fetch`](https://github.com/algorand-devrel/x402-demo) which delegates to the wallet adapter, so Pera works out of the box once the user has an opted-in account.

### ParsecConnect — the canonical browser↔wallet bridge

The live AgenticPlace deployment ships **`ParsecConnect`** v1.1.0 ([`/home/hacker/live/modules/parsec-connect.js`](file:///home/hacker/live/modules/parsec-connect.js), © 2026 BANKON / AgenticPlace, GPL-3.0) — a drop-in browser SDK that talks to a locally-running Parsec wallet via WebSocket. This is the canonical web-side bridge and is **Pera-compatible by design**: the `signTransaction([[{ txn, signers: [addr] }]], message)` shape matches Pera Connect, so dApps swap between them without code changes.

```js
const parsec = new ParsecConnect({ debug: true });
const status = await parsec.isAvailable();
if (status.available && status.unlocked) {
  const accounts = await parsec.connect();
  const signed = await parsec.signTransaction(
    [[{ txn, signers: [addr] }]],
    'x402 payment for /weather',
  );
}
```

Surface:

| Aspect | Value |
|---|---|
| Default port | `9876` (configurable) |
| Health endpoint | `GET http://localhost:9876/parsec/v1/connect/health` |
| WebSocket | `ws://localhost:9876/parsec/v1/connect/ws` |
| RPC methods | `parsec_accounts`, `parsec_signTransactions`, `ping` (heartbeat) |
| Error codes | `USER_REJECTED 4001`, `WALLET_LOCKED 4100`, `TIMEOUT 4002`, `DISCONNECTED 4003`, plus JSON-RPC `-32700/-32601/-32602` |
| Reconnection | Exponential backoff, max 3 attempts, 30 s heartbeat |
| Sign timeout | 120 s default (long enough for human approval in the Parsec UI) |
| Origin attestation | Browser `window.location.origin` is included in every sign request — Parsec's UI shows where the request came from |

For mindX's web frontend (and AgenticPlace's existing dashboards under `/home/hacker/live/`), `ParsecConnect` is the recommended path for any x402 flow that needs human-in-the-loop approval. The `bridge.ts` documented above is the **autonomous-agent** path (mnemonic in vault, no UI prompt); `ParsecConnect` is the **interactive-human** path (vault stays in the desktop app, web sees only the signed bytes).

### Parsec Wallet

**Parsec Wallet** (`/home/hacker/parsec/parsec-wallet/`, © 2026 BANKON, GPL-3.0) is a **sister BANKON project** — same `cypherpunk2048` lineage as mindX, sharing the BANKON Vault crypto stack, the pgvectorscale memory tier, and the cypherpunk operator philosophy. It is the canonical Algorand wallet for the mindX/AgenticPlace ecosystem and ships **a full x402 client implementation** — not a third-party integration but a co-developed surface.

Architecture: Tauri 2.0 desktop app (Rust backend + vanilla TS / Blueprint.js frontend), `algosdk@^3.0.0` and `@algorandfoundation/algokit-utils@^9.2.0`, encrypted key storage via the **`bankon_vault` Rust module** (Argon2id + AES-256-GCM — the same crypto family as mindX's [`mindx_backend_service/bankon_vault/`](../mindx_backend_service/bankon_vault/)).

The x402 module lives at `src/lib/x402/` (8 files, three integration tiers):

| File | Tier | Role |
|---|---|---|
| `types.ts` | 1 — direct import | ERC-8004 `AgentRegistration`, `CommandChannel`, `BankonPaymentRequirement`, `AccessTier` (Visitor → Imperator), payment manifest types |
| `constants.ts` | 1 | BANKON ASA `203977300`, treasury address `44FM64A7UOXRVCM66TOHOATSQK3WXAXISQC6KC3OIVSKARTBSLN2DIJ3YI`, ERC-8004 CREATE2 addresses (15+ EVM chains), [CAIP-2](https://chainagnostic.org/CAIPs/caip-2) network IDs |
| `oracle.ts` | 3 — HTTP | `PriceOracle` via [Vestige DEX](https://free-api.vestige.fi) (free, no API key) for ALGO/USD + arbitrary ASA prices |
| `discount.ts` | 3 | `checkBankonHolder` + `applyDiscount` — BANKON ASA holders get **50% off x402 fees** (5-min cache per address) |
| `bridge.ts` | 2 — vault-secured | `buildAlgorandX402Signer(address, passphrase, network)` — retrieves mnemonic from Rust vault, builds algosdk signer, **secret lives in JS closure for one signing cycle then GC'd**. Plus `signBytesWithVault`, `sendPaymentWithVault` |
| `payment.ts` | 2 | `x402Fetch(url, init, address, passphrase, network, approveCallback)` — full 402 → pay → settle flow with user-approval hook |
| `agenticplace-client.ts` | 3 | HTTP client for `agenticplace.pythai.net` (discovery), `mindx.pythai.net` (paywalled endpoints), `mindx.pythai.net:4022` (facilitator), `bankon.pythai.net` (identity) |
| `index.ts` | — | Barrel export |

**Six access tiers** by BONA FIDE balance (`types.ts`): Visitor (0), Citizen (IDNFT only), Artisan (100 BONA FIDE), Senator (1000), Consul (5000), Imperator (10000+). Tiers gate command-channel access on the agent registry.

**Vault-secured signing is the security innovation.** `keystoreRetrieve` pulls the mnemonic from the Rust `bankon_vault` (AES-256-GCM at rest), `algosdk.mnemonicToSecretKey` derives the signing key into a closure, the signer interface uses it for one x402 cycle, then the closure goes out of scope and the secret is GC-eligible. The vault is the only persistent secret store; JS never serializes the key. See [`src/lib/x402/bridge.ts`](file:///home/hacker/parsec/parsec-wallet/src/lib/x402/bridge.ts).

The vault key documented below (`algorand_mnemonic` for autonomous mindX backend flows) **mirrors the same custody pattern** Parsec uses for desktop signing — both encrypt to AES-256-GCM, both apply ephemeral-key discipline, both treat the mnemonic as the only persistent secret. mindX's BANKON Vault is the server-side counterpart of Parsec's `bankon_vault`.

### Multisig (Algorand-native)

For boardroom-tier or treasury-tier payments where one signer is not enough:

```bash
# 2-of-3 multisig from three pera-derived keys
goal account multisig new -T 2 ADDR1 ADDR2 ADDR3
# Sign a partial transaction
algokey multisig --keyfile=signer1.key --txfile=tx.txn
# Merge two signed copies before submission
goal clerk multisig merge -o tx.merged.txn tx.signer1.txn tx.signer2.txn
```

mindX should expose a `tools/x402_avm_multisig_client.py` (deferred) that wraps `tools/x402_avm_client.py` with a multisig descriptor instead of a single mnemonic — the rest of the 402 flow is unchanged.

## Algorand DevRel's `@x402-avm/*` — respectful integration

Algorand DevRel published `@x402-avm/{core,avm,hono,fetch}` v^2.6.1 (ISC, 2026-05-02) — chain-agnostic envelope types + a Hono server middleware + a `wrapFetchWithPayment` client wrapper + the AVM signing helpers. Repo: [`algorand-devrel/x402-demo`](https://github.com/algorand-devrel/x402-demo). Reference hosted facilitator: [`x402.goplausible.xyz`](https://x402.goplausible.xyz) (operated by GoPlausible) — alternative to the mindX-hosted canonical at `mindx.pythai.net:4022`. Status: alpha (depends on `@algorandfoundation/[email protected]`), TestNet-only, ships **no smart contracts** — settlement is a plain `AssetTransferTxn`.

mindX consumes these libs as **HTTP glue** for the AVM rail:

- [`AgenticPlace/api/x402_avm.ts`](../AgenticPlace/api/x402_avm.ts) — wraps `@x402-avm/{fetch,hono,avm,core}` for in-browser and Node consumers
- [`tools/x402_avm_client.py`](../tools/x402_avm_client.py) — the Python equivalent (does not depend on the npm packages — pure `py-algorand-sdk`, mirroring `keeperhub_x402_client.py`)

The contracts mindX is shipping (the `X402Receipt` pair) **complement** rather than replace this work — they sit on-chain where DevRel's libs explicitly stop. The Algorand DevRel team's due diligence on the chain-agnostic envelope shape, the hosted facilitator pattern, and the alpha-stage HTTP wiring is exactly the kind of industry-standard groundwork that makes mindX's contract layer slot in cleanly. Done correctly, the canonical `X402Receipt` pattern flows back into the ecosystem as a reference for any project that wants on-chain receipt parity.
